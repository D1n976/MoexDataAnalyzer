# MOEX Data Analyzer

Веб-приложение для анализа исторических котировок акций Московской биржи (MOEX). Позволяет просматривать графики цен нескольких инструментов одновременно, сравнивать их статистику за выбранный период и экспортировать результаты в PDF.

## Архитектура

Приложение состоит из трёх сервисов, запускаемых через Docker Compose:

```
┌─────────────────────────────────────────────────────────┐
│                     docker-compose                      │
│                                                         │
│  ┌──────────────┐    ┌──────────────┐  ┌─────────────┐ │
│  │   frontend   │    │     app      │  │     db      │ │
│  │  Nginx :80   │───▶│ Spring :8081 │──│ Postgres:5432│ │
│  └──────────────┘    └──────────────┘  └─────────────┘ │
│         │                   │                           │
└─────────┼───────────────────┼───────────────────────────┘
          │                   │
      Браузер           MOEX ISS API
                    (iss.moex.com)
```

| Компонент  | Технология                  | Порт |
|------------|-----------------------------|------|
| Frontend   | React 19 + Vite + Nginx     | 80   |
| Backend    | Spring Boot 4.0.3 + Java 25 | 8081 |
| Database   | PostgreSQL 15               | 5432 |

### Аутентификация и авторизация

Используется **сессионная аутентификация** Spring Security (HttpSession + SecurityContext).

**`SecurityConfig`** настраивает:
- CSRF отключён (SPA без форм не использует его)
- HTTP Basic отключён (чтобы браузер не показывал нативный диалог логина)
- Пути `/api/auth/**` открыты без аутентификации, все остальные — требуют авторизованной сессии
- CORS: разрешённые origin берутся из переменной окружения `CORS_ORIGIN`
- При 401 возвращается JSON `{"error":"Unauthorized"}` вместо redirect

**Модель пользователя (`User`)** реализует `UserDetails` и хранится в таблице `users`:
- `id` — автоинкремент
- `name` — уникальное имя пользователя
- `email` — уникальный email
- `pass` — пароль, зашифрованный BCrypt

### Регистрация с верификацией email

Процесс регистрации двухшаговый:

```
Шаг 1: POST /api/auth/register
  ├── Проверка совпадения паролей
  ├── Проверка уникальности email (UserRepository.existsByEmail)
  ├── Проверка уникальности имени (UserRepository.existsByName)
  ├── Генерация 6-значного кода (Random, 000000–999999)
  ├── Сохранение {code, expiry (+10 мин), userDto} в ConcurrentHashMap
  └── Отправка кода на email через JavaMailSender (SMTP, STARTTLS)

Шаг 2: POST /api/auth/verify
  ├── Поиск записи в pending-map по email
  ├── Проверка срока действия (10 минут)
  ├── Сравнение кода
  ├── Создание пользователя в БД (registerUser)
  └── Удаление записи из pending-map
```

**Важная деталь:** pending-запись удаляется только после успешного сохранения в БД. Если `registerUser()` выбросит исключение (например, дубликат в БД), код остаётся валидным и пользователь может повторить попытку.

### Управление аккаунтом

| Метод  | Путь                | Описание                                              |
|--------|---------------------|-------------------------------------------------------|
| POST   | `/api/auth/login`   | Аутентификация, создание сессии                       |
| POST   | `/api/auth/register`| Шаг 1: проверки + отправка кода на email             |
| POST   | `/api/auth/verify`  | Шаг 2: верификация кода + создание пользователя       |
| DELETE | `/api/auth/account` | Удаление аккаунта текущего пользователя + инвалидация сессии |

### Работа с MOEX ISS API

**`MoexService`** получает исторические данные с публичного API Московской биржи:

```
https://iss.moex.com/iss/history/engines/stock/markets/shares/securities/{secid}.json
  ?from=YYYY-MM-DD&till=YYYY-MM-DD&start=N
```

API возвращает максимум **100 записей за запрос**. Для получения полного диапазона реализована **пагинация через `history.cursor`**:

```
Цикл:
  1. Запрос с параметром start=N (начало с 0)
  2. Парсинг cursor: {INDEX, TOTAL, PAGESIZE}
  3. Если INDEX + PAGESIZE >= TOTAL → данные кончились, выход
  4. Иначе start = INDEX + PAGESIZE → следующая страница
```

Список доступных инструментов хранится в файле `backend/src/main/resources/static/tickers` (JSON). Читается через `ClassPathResource.getInputStream()` — это позволяет работать внутри JAR-архива.

**Модель свечи (`MoexStockCandle`):**
| Поле        | Описание              |
|-------------|-----------------------|
| `TRADEDATE` | Дата торгов           |
| `NAME`      | Краткое название      |
| `OPEN`      | Цена открытия         |
| `HIGH`      | Максимум дня          |
| `LOW`       | Минимум дня           |
| `CLOSE`     | Цена закрытия         |
| `VOLUME`    | Объём торгов          |

### Аналитика

**`MoexAnalysisService`** вычисляет статистику по массиву свечей:

| Метрика              | Формула                                              |
|----------------------|------------------------------------------------------|
| Средняя цена         | Среднее арифметическое CLOSE за период               |
| Максимум             | max(CLOSE) за период                                 |
| Минимум              | min(CLOSE) за период                                 |
| Объём торгов         | sum(VOLUME) за период                                |
| Изменение цены       | CLOSE(последний день) − CLOSE(первый день)           |
| Изменение в %        | (lastClose − firstClose) / firstClose × 100          |
| Торговых дней        | Количество записей за период                         |

### API эндпоинты котировок

| Метод | Путь                      | Параметры               | Описание                         |
|-------|---------------------------|-------------------------|----------------------------------|
| GET   | `/stock/api/getTickers`   | —                       | Список доступных тикеров         |
| GET   | `/stock/api/getCandles`   | `secid`, `from`, `till` | Исторические свечи по инструменту|
| GET   | `/stock/api/analyzeCandles`| `secid`, `from`, `till`| Статистика по инструменту        |

## Фронтенд

### Стек

- **React 19** — UI-компоненты
- **Vite** — сборка
- **Recharts** — графики
- **Axios** — HTTP-запросы с `withCredentials: true` (передача cookie сессии)
- **html2canvas + jsPDF** — экспорт в PDF (ленивая загрузка)
- **Nginx** — статический сервер + reverse proxy

### Маршрутизация

```
/          → редирект на /login
/login     → форма входа/регистрации (открыта)
/dashboard → дашборд (защищённый роут, требует isAuthenticated)
*          → редирект на /login
```

Состояние аутентификации хранится в `localStorage` (`isAuthenticated: true/false`). При 401/403 от сервера выполняется автоматический logout.

### Компоненты

**`Login`** — форма входа и регистрации:
- Переключение режимов (вход / регистрация) без перезагрузки
- При регистрации после успешной отправки показывает экран ввода 6-значного кода
- Ошибки отображаются inline (красный блок), без браузерных `alert()`
- После успешной верификации — зелёное сообщение и переход на форму входа

**`Dashboard`** — основной экран:
- **Сайдбар** (левая панель):
  - Список тикеров с чекбоксами для мультиселекта
  - Выбор диапазона дат (от/до)
  - Кнопка «Загрузить данные»
  - Переключатель вида: «Совмещённый» / «Раздельный» (появляется после загрузки)
  - Кнопка «Скачать PDF» (появляется после загрузки)
- **Основная область**:
  - Пустое состояние с подсказкой, пока данные не загружены
  - **Совмещённый режим**: все выбранные тикеры на одном графике, разными цветами
  - **Раздельный режим**: отдельный график для каждого тикера
  - Карточки статистики под графиками
- **Хедер**:
  - Кнопка «Выход»
  - Кнопка «Удалить аккаунт» с inline-подтверждением («Вы уверены? / Да, удалить / Отмена»)
  - Баннер уведомлений (ошибки/успех, автоскрытие через 4 сек)

**`StatCard`** — карточка статистики по тикеру:
- Тикер, значок % изменения (зелёный/красный)
- Средняя, максимальная, минимальная цена, абсолютное изменение, количество торговых дней

**`CustomTooltip`** — кастомный тултип на графике с датой и ценами всех выбранных инструментов.

### Загрузка данных

При нажатии «Загрузить данные» делаются **параллельные запросы** для всех выбранных тикеров:

```js
Promise.all([
  // Свечи для всех тикеров
  Promise.all(selected.map(t => axios.get('/stock/api/getCandles', {params: {secid: t, from, till}}))),
  // Статистика для всех тикеров
  Promise.all(selected.map(t => axios.get('/stock/api/analyzeCandles', {params: {secid: t, from, till}}))),
])
```

Затем данные объединяются по дате в формат `[{date, SBER: 280.5, GAZP: 150.2, ...}]` для Recharts.

### Экспорт в PDF

`html2canvas` делает скриншот области с графиками и статистикой (scale=2 для чёткости), затем `jsPDF` сохраняет его как PDF в альбомной ориентации A4. Имя файла: `moex_{тикеры}_{от}_{до}.pdf`.

## Взаимодействие компонентов

```
Браузер (React)                Nginx                  Spring Boot
      │                          │                          │
      │  GET /                   │                          │
      │─────────────────────────▶│  index.html + JS бандл  │
      │◀─────────────────────────│                          │
      │                          │                          │
      │  POST /api/auth/login    │  proxy_pass :8081        │
      │─────────────────────────▶│─────────────────────────▶│
      │  Set-Cookie: JSESSIONID  │                          │
      │◀─────────────────────────│◀─────────────────────────│
      │                          │                          │
      │  GET /stock/api/getTickers (с cookie)               │
      │─────────────────────────▶│─────────────────────────▶│
      │  ["SBER","GAZP",...]     │                          │
      │◀─────────────────────────│◀─────────────────────────│
      │                          │                          │
      │  GET /stock/api/getCandles?secid=SBER&from=...      │
      │─────────────────────────▶│─────────────────────────▶│──▶ MOEX ISS API
      │  [{TRADEDATE, CLOSE,...}]│                          │◀── (с пагинацией)
      │◀─────────────────────────│◀─────────────────────────│
```

Nginx проксирует:
- `/api/*` → `http://app:8081/api/`
- `/stock/api/*` → `http://app:8081/stock/api/`

Все остальные запросы отдают `index.html` (SPA fallback).

## Запуск

### Требования

- [Docker](https://docs.docker.com/get-docker/) + [Docker Compose](https://docs.docker.com/compose/install/)

### 1. Клонирование репозитория

```bash
git clone https://github.com/D1n976/MoexDataAnalyzer.git
cd MoexDataAnalyzer
```

### 2. Настройка переменных окружения

Скопируйте шаблон и заполните значения:

```bash
cp .env.example .env
```

Откройте `.env` и укажите:

```env
# База данных
DB_USERNAME=postgres
DB_PASSWORD=ваш_пароль

# SMTP для отправки кодов верификации
SMTP_HOST=smtp.mail.ru          # или smtp.gmail.com и т.д.
SMTP_PORT=587
SMTP_USERNAME=ваш@email.ru
SMTP_PASSWORD=пароль_приложения
SMTP_FROM=ваш@email.ru
```

> **Важно:** Для Gmail и Mail.ru нужен **пароль приложения**, а не пароль от аккаунта.
> - Mail.ru: Настройки → Безопасность → Пароли для внешних приложений
> - Gmail: Аккаунт → Безопасность → Двухэтапная аутентификация → Пароли приложений

### 3. Запуск

```bash
docker-compose up --build
```

После сборки приложение доступно по адресу: **http://localhost**

Первый запуск занимает несколько минут — идёт загрузка образов и сборка Java/Node.

### Управление

| Команда | Описание |
|---------|----------|
| `docker-compose up --build` | Запустить с пересборкой образов |
| `docker-compose up -d --build` | Запустить в фоновом режиме |
| `docker-compose down` | Остановить (данные БД сохраняются) |
| `docker-compose down --volumes` | Остановить и **удалить** данные БД |
| `docker-compose logs -f app` | Логи бэкенда в реальном времени |
| `docker-compose logs -f frontend` | Логи фронтенда/nginx |

### Пересборка отдельного сервиса

```bash
# Только бэкенд
docker-compose build app && docker-compose up -d app

# Только фронтенд
docker-compose build frontend && docker-compose up -d frontend
```
